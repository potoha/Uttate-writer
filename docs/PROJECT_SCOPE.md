# Uttate MVP プロジェクトスコープ

## 1. 文書の目的

本書は `project.md` を実装可能な境界へ落とし込み、Uttate の最初のMVPで「作るもの」「作らないもの」「完成とみなす条件」を固定する。

## 2. プロダクトの目的

Uttate は、ローマ字・ひらがな・英単語・誤字・表記揺れが混ざったラフ入力を、思考を止めずに入力した後で自然な日本語へ変換し、候補を確認して採用できるスタンドアロンのデスクトップアプリである。

MVPで検証する中心仮説は次のとおり。

> ユーザーが変換完了を待たずに次の思考チャンクを書き続けられ、後から候補を比較・採用できれば、Spaceキーによる逐次変換よりも思考の流れを保ちやすい。

## 3. 対象ユーザーと利用場面

### 対象ユーザー

- 日本語の文章を長めに書く人
- ローマ字・英語混じりのラフ入力から文章を組み立てたい人
- 変換候補の選択で思考が中断されることを負担に感じる人
- LM StudioなどのローカルLLMを利用できる試験ユーザー

### 主な利用場面

- アイデア、メモ、下書きを勢いを保ったまま入力する
- 入力後にチャンク単位で変換候補を比較する
- 曖昧な固有名詞や表記を原文と照合して修正する
- 採用済みチャンクを一つの文章としてクリップボードまたはファイルへ書き出す

## 4. MVPの成果物

MVPは Python 3.11以降と PySide6 を使うデスクトップアプリとして提供する。成果物には次を含める。

1. キーボード中心で操作できる2ペインUI
2. チャンク単位の非同期変換キュー
3. `Chunk` / `Document` データモデルと状態遷移
4. LM Studio・OpenAI互換・MockのLLMプロバイダー抽象化
5. 読み正規化、辞書検索、漢字かな変換からなる変換パイプライン
6. SQLiteによるローカル辞書と明示的なユーザー辞書登録
7. 候補A・候補B・原文の比較、採用、手動編集、再変換
8. クリップボードおよび `.txt` / `.md` への書き出し
9. 設定ファイル、サンプル辞書、テスト用入力
10. 単体テストと主要ユーザーフローの結合テスト

## 5. 機能スコープ

### 5.1 入力とチャンク管理

- 大きなラフ入力欄を提供する。
- `Enter` で現在の入力をチャンクとして確定し、変換キューへ送る。
- `Shift+Enter` で送信せずに改行する。
- 空白のみの入力はチャンクとして登録しない。
- 送信直後に入力欄を空け、変換中でも次の入力を受け付ける。
- チャンク一覧に状態、短い内容、選択状態を表示する。
- 失敗したチャンクは原文を保持し、再実行できる。

### 5.2 変換パイプライン

各チャンクは以下の順で処理する。

1. Stage 1: LLMによる読み正規化
2. Stage 2: 正規化結果を用いたローカル辞書候補検索
3. Stage 3: LLMによる候補A・候補Bの生成
4. Review: 候補、原文、不確実箇所をユーザーへ提示

変換には元入力、正規化された読み、関連辞書候補、直前までの採用済み文章を使用する。候補Aは忠実性を優先し、候補Bは意味を追加しない範囲で自然さを優先する。

### 5.3 LLMプロバイダー

- 共通の `complete_json` インターフェースを設ける。
- LM Studioの既定URLは `http://localhost:1234/v1` とする。
- モデル名、URL、APIキーを設定から変更できるようにする。
- モデル名はコードへ固定しない。
- Mockプロバイダーはネットワークなしで決定的な結果を返す。
- 不正JSON、タイムアウト、接続失敗をユーザーへ表示し、アプリ本体は継続動作する。

### 5.4 辞書

- SQLiteに共通・ユーザー・プロジェクト・最近採用した表記の由来を保持できる構造を用意する。
- 完全一致と長いn-gramを優先して検索する。
- 優先順はユーザー、プロジェクト、最近の採用、共通辞書を基本とする。
- 1語あたり5候補、最大30語、全体80候補を上限とする。
- MVPでの辞書登録はユーザーの明示操作時のみ行う。
- 辞書は表記候補の供給に限定し、知識検索やRAGとして扱わない。

### 5.5 レビュー

- 原文、正規化結果、現在の候補、不確実箇所を表示する。
- `Tab` / `Shift+Tab` で「候補A → 候補B → 原文」を循環表示する。
- `Enter` で表示中の候補を採用する。
- `Ctrl+Enter` で採用して次の未解決チャンクへ移動する。
- `E` で現在の候補をその場で編集する。
- `R` で選択中チャンクを再変換する。
- `Esc` で未確定のプレビューまたは編集を取り消す。
- 不確実性を隠さず、該当箇所、候補、理由を表示する。

### 5.6 書き出し

- チャンク順に `adopted_text` を連結する。
- 未採用チャンクは設定に従い候補Aまたは原文へフォールバックする。
- 最終文章をクリップボードへコピーできる。
- UTF-8の `.txt` または `.md` として保存できる。

### 5.7 設定

- プロバイダー種別、URL、APIキー、モデル名を変更できる。
- 辞書利用、候補数、未採用時の書き出し規則を変更できる。
- APIキーはログや画面上の通常表示へ露出させない。
- MVPでは設定をローカルJSONへ保存する。

## 6. MVP対象外

以下はMVPへ含めない。

- OSレベルのIME、任意アプリへのグローバル文字置換
- モバイル版、Web版、ブラウザー拡張
- ユーザーアカウント、クラウド同期、共同編集
- RAG、Web検索、一般的な文章生成チャット
- プラグイン市場や外部拡張API
- モデルの学習、LoRA/QLoRA、独自推論ランタイム
- 自動辞書学習やユーザー確認なしの語彙登録
- 高度な校正、事実確認、文章内容の創作
- 候補の品質最適化を目的とした大規模評価基盤
- OSインストーラー、コード署名、自動更新（MVP完了後の配布フェーズで扱う）

## 7. 非機能要件

### 応答性

- `Enter` 操作から次の入力を開始できる状態まで、ローカルUI処理は体感上即時であることを目標とする（目安100ms以内）。
- LLM通信と辞書検索をUIスレッドで実行しない。
- 複数チャンクの処理中も入力、選択、スクロールを妨げない。

### 信頼性

- 1チャンクの失敗が他のチャンクやアプリ全体を停止させない。
- 状態遷移を明示し、処理中・レビュー待ち・失敗を区別する。
- 再変換時も原文と最後に採用した文章を失わない。

### 忠実性

- 入力にない主張や情報を追加しない。
- 候補Aは意味保存を最優先する。
- 判断不能な表記は無理に漢字化せず、不確実性として残す。

### テスト容易性

- UI、パイプライン、プロバイダー、ストレージを分離する。
- Mockプロバイダーでネットワークを使わず主要フローを再現できる。
- 一時SQLiteデータベースを使って辞書検索をテストできる。

### ローカル性

- 辞書、設定、文書データはローカルに保持する。
- 外部通信はユーザーが選択したLLMプロバイダーへの要求に限定する。

## 8. 技術境界

- UI: PySide6
- 言語: Python 3.11+
- HTTP: OpenAI互換のChat Completionsを扱えるクライアント
- 永続化: SQLite（辞書と最小限の文書状態）、JSON（設定）
- テスト: pytest、必要に応じて pytest-qt
- パッケージ構成: `src/uttate/` の小文字パッケージ名を採用する
- 非同期実行: Qtのシグナル/スロットとワーカースレッドまたはスレッドプールを利用し、UI更新はUIスレッドで行う

## 9. 主要な状態遷移

```text
raw
  → normalizing
  → normalized
  → retrieving_dictionary
  → converting
  → ready_for_review
  → adopted | edited

各処理状態 → failed → 再変換で normalizing
```

再変換中に古い応答が後着した場合は、処理IDまたは世代番号で古い結果を破棄する。

## 10. MVP受け入れ基準

以下をすべて満たしたときMVP完成とする。

1. ユーザーがラフ入力を `Enter` で3チャンク以上連続送信でき、先行チャンクの変換完了を待たずに次を入力できる。
2. Mockプロバイダーで各チャンクが `raw` から `ready_for_review` まで遷移し、UIに反映される。
3. LM Studio設定時にStage 1とStage 3がOpenAI互換API経由で実行される。
4. サンプル辞書の語が正規化された読みから検索され、Stage 3へ関連候補だけが渡される。
5. レビュー画面で候補A、候補B、原文を循環し、採用・編集・再変換ができる。
6. 不確実性が存在する場合、ユーザーが画面上で確認できる。
7. 採用済み文章をクリップボード、`.txt`、`.md` のいずれにも出力できる。
8. 接続失敗や不正JSONが起きても入力済みデータを失わず、該当チャンクを再実行できる。
9. `project.md` の初期テスト入力5件を手動確認でき、Mockテストでは決定的な期待値を検証できる。
10. 主要な単体・結合テストがローカルで成功し、READMEに起動・設定・テスト方法が記載されている。

## 11. スコープ判断の原則

実装中に追加案が出た場合は、次の順で判断する。

1. 「入力を止めない → 後からレビューする」の検証に必須か。
2. 受け入れ基準の達成に必要か。
3. Mockで再現可能な小さな実装か。
4. 必須でなければMVP後のバックログへ送る。

モデル精度の改善より、まず一連のUXループが途切れず動くことを優先する。
